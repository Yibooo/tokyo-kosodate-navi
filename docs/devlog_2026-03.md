# 開発日記 #1｜東京23区・子育て支援ナビゲーターを作った話

> 2026年3月｜個人開発 / Next.js / Supabase / Claude API

---

## なぜ作ったか

子育て支援制度って、意外と複雑だ。

国が出している児童手当、東京都の018サポート、各区独自のお祝い金や医療費助成——同じ子どもを持つ親でも、住んでいる区が違えば受け取れる額が大きく変わる。知っているか知らないかで、年間で数十万円の差が出ることもある。

でも、それを一元的に調べられるサービスがない。区のホームページはどこも見づらいし、国と都と区を横断して「自分がもらえる制度の総額」を計算できるツールは存在しない。

そこで作ることにした。**「東京23区・子育て支援ナビゲーター」** だ。

居住区・子の年齢・世帯年収を入力するだけで、受給できる制度を自動判定して一覧表示する。申請漏れをなくし、もらえるお金を最大化するためのツール。

---

## 技術構成

シンプルに行くと決めた。

- **Frontend / Backend**：Next.js 16（App Router）
- **DB**：Supabase（PostgreSQL）
- **AI**：Claude API（スクレイピング時のHTMLデータ抽出のみ）
- **ホスティング**：Vercel
- **通知**：Resend（メール）

最初の設計で一番悩んだのが、**マッチングエンジンをLLMにするかどうか** だった。

ユーザーが条件を入力したとき、「あなたに合う制度を探す」ロジックとして、Claude APIをそのまま使う方法もある。でも、それだと毎回APIを呼ぶ分コストがかかるし、レスポンスも遅い。

最終的に選んだのは **「スクレイピング時だけLLMを使い、検索時はDB直接マッチング」** というアーキテクチャだ。

```
[管理者] スクレイピング実行
      ↓
   Claude API（HTMLを構造化JSONに変換）
      ↓
   Supabase に保存（ドラフト）
      ↓
   管理者がレビュー・承認
      ↓
   承認済みデータのみ本番に反映

[ユーザー] 条件を入力
      ↓
   DBから承認済み制度を取得
      ↓
   ルールベースで年齢・所得・居住区をマッチング
      ↓
   結果を表示
```

LLMはデータ収集の時だけ。ユーザーが検索するたびにAPIを叩くことはしない。これによって、**検索は1秒以内・コストゼロ・スケーラブル** になった。

## Phase 0〜2 を振り返る

### Phase 0：土台を作る

まず DB のスキーマを設計した。`policies`（制度テーブル）と `policy_conditions`（条件テーブル）を分けた設計にした。条件の複雑さをテーブルで表現するのは最初少し迷ったが、分離しておいてよかった。

スクレイピングパイプラインは、ターゲットURLを順番に処理して、HTMLをClaudeに渡し、構造化JSONを受け取る形。抽出したデータはいったん `status='draft'` で保存して、管理者がレビューしてから `approved` にする。直接本番に流れる作りにしなかったのは、AIの誤抽出を防ぐためだ。

管理者レビュー画面（`/admin/review`）も最初から作った。Webベースで承認・却下できる画面があると、後の運用がはるかに楽になる。

### Phase 1：検索機能を作る

入力フォームは4項目にしぼった。

- 居住区（ドロップダウン）
- 子の生年月日 または 出産予定日（年月セレクター）
- 第何子か
- 世帯年収（スライダー）

マッチングロジックは `src/lib/matcher.ts` に集約した。年齢・所得・出生順位・居住区の4条件でフィルタリングして、月額換算でソートする。シンプルだが、これで十分機能する。

結果表示は、国 / 東京都 / 区 の層別フィルタータブと、カテゴリフィルターを組み合わせた構成にした。

### Phase 2：一気に拡張する

Phase 1 が動いたところで、データの少なさが気になってきた。最初は5区・14制度しかなかった。

思い切って一気に拡張することにした。

**23区すべてに対応**し、スクレイピングターゲットを 12件 → 33件 に増やした。国・都・区ごとに重要な制度を手動シードとして登録し、承認済み制度数は **67件** になった。

登録した制度の中でも特に注目なのは：

- **渋谷区の保育料完全無料化**（0〜5歳・所得制限なし・全国初）
- **乳児養育手当**（江戸川区・月13,000円）
- **出産祝い金**（足立区・第3子は30万円）
- **電動アシスト自転車購入費補助**（世田谷区・練馬区など複数区）
- **住宅ローン控除の子育て特例**（国・控除率0.7%）

こういう制度、知らない人が多いと思う。

制度には8つのカテゴリを設定して、絞り込みできるようにした。

| カテゴリ | 例 |
|---------|---|
| 💰 給付金・手当 | 児童手当、乳児養育手当 |
| 🎁 一時金 | 出産育児一時金 |
| 🏥 医療費助成 | 子ども医療費助成（高校生まで無料） |
| 🏫 保育・教育 | 幼児教育無償化 |
| 👶 育児サービス | ベビーシッター補助 |
| 💼 雇用・休業 | 育児休業給付金 |
| 📊 税制優遇 | 住宅ローン控除 |
| 🚲 物品・生活支援 | 電動自転車補助 |

管理者画面も大幅強化した。

- 統計ダッシュボード（承認待ち・承認済み・却下済みの件数）
- 一括承認ボタン（まとめて全件承認できる）
- スクレイピングログ閲覧タブ（いつ何件取得したかの履歴）

## 詰まったところ

### ビルドエラー：同名の `const` が2回定義されていた

Phase 2 の実装を終えて `git push` したとき、Vercel のビルドが失敗した。

```
Identifier 'LAYER_LABELS' has already been declared.
```

原因は、`admin/review/page.tsx` で同じ `const` を2回宣言していたこと。ファイルの先頭に定義したものを、後半にもう一度書いてしまっていた。

修正は数行の削除で済んだ。ただ、ビルドエラーが本番デプロイ後に発覚するのは地味にヒヤリとする体験だった。今後はローカルで `next build` を確認してからプッシュする習慣をつけたい。

## 次にやること（Phase 3）

- **Cron 自動更新**：月1回の自動スクレイピングを Vercel Cron で設定する

## 所感

個人開発で一番楽しいのは、設計の決断を全部自分でできること。

LLMをどこに使うか、DBをどう設計するか、フィルターのUXをどう作るか——チームで開発していれば議論に時間がかかる部分を、自分の判断でさっさと実装できる。

もちろん、間違える。ビルドエラーもあった。でも間違えたら直せばいい。

子育て支援制度は、知っているか知らないかで人生が変わる類のものだと思う。このツールが誰かの申請漏れをひとつでも防げたら、それで十分意味がある。
